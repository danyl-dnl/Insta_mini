2. The Generator — get_db()
The most important concept — where generator + async meet
What is a Generator? Generator
A generator is a Python function that uses yield instead of return. It pauses at yield, hands you a value, and resumes from that point later. This is perfect for "setup → use → cleanup" patterns.
concept example
# Normal function: returns and is completely done
def normal():
    return 42

# Generator: pauses at yield, resumes after
def gen():
    print("before yield")
    yield 42       # pauses here — caller gets 42
    print("after yield")  # runs after caller is done
get_db() — The Real Code from Your Project Generator
This is the actual get_db function in Setting/Database.py. It creates a DB session, yields it to the endpoint, and automatically closes it when done — even if an error happens.
Setting/Database.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker
from sqlalchemy.orm import declarative_base
from Setting.Config import DATABASE_URL

# 1. Async engine = the connection to the database
engine = create_async_engine(DATABASE_URL, echo=True)

# 2. Base class — all table models inherit from this
Base = declarative_base()

# 3. Session factory — creates AsyncSession objects
SessionLocal = async_sessionmaker(bind=engine, expire_on_commit=False)

# 4. THE GENERATOR ← This is what mentors will ask about!
async def get_db():
    async with SessionLocal() as session:
        yield session   # PAUSE: give session to the endpoint
                        # RESUME: auto-closes when endpoint is done
💡
Memory trick: Think of get_db like a tap. async with SessionLocal() opens the tap, yield session lets water flow to your endpoint, and when the endpoint is done the tap closes automatically — you never forget to close it!

5. async / await — Non-Blocking Code
Why everything has async and await in front of it
Why async? async
When your server asks the database for data, it takes time (50-100ms). With synchronous code, the server just sits and waits, blocking all other requests. With async/await, the server goes off and handles other requests while waiting for the DB. It's like a waiter serving multiple tables instead of standing at one waiting for food.
User/CRUD/user.py — real async examples
# Rule 1: Any function that uses await must be async
async def create_user(db: AsyncSession, username, email, password, ...):
    new_user = models.User(username=username, email=email,
                             password=hash_password(password),
                             created_at=datetime.utcnow())
    db.add(new_user)
    await db.commit()            # await: waits for DB to save
    await db.refresh(new_user)  # await: waits for DB to return generated id
    return new_user

# Rule 2: To call an async function you must also await it
async def get_user_by_username(db: AsyncSession, username):
    result = await db.execute(
        select(models.User).where(models.User.username == username)
    )
    return result.scalar_one_or_none()

A Pydantic model is a Python class that validates JSON data automatically. When a user sends a request body, FastAPI uses the schema to check that all required fields are present and have the correct types. An ORM lets you work with database tables as if they were Python classes. Instead of writing raw SQL like SELECT * FROM users WHERE user_id = 1, you write Python: select(User).where(User.user_id == 1). SQLAlchemy translates it to SQL for you.

8. Authentication — JWT Tokens
How login works: password hashing, JWT, refresh tokens
Step 1: Password Hashing with bcrypt
Passwords are never stored as plain text. They are hashed using bcrypt — a one-way algorithm. You cannot reverse a hash to get the original password; you can only compare a plain password against the stored hash.
Setting/Security.py — actual code
import bcrypt

def hash_password(password: str) -> str:
    pwd_bytes = password.encode("utf-8")   # string → bytes
    salt = bcrypt.gensalt()               # random salt added to hash
    hashed = bcrypt.hashpw(pwd_bytes, salt)
    return hashed.decode("utf-8")         # bytes → string for DB storage

def verify_password(plain_password: str, hashed_password: str) -> bool:
    try:
        return bcrypt.checkpw(
            plain_password.encode("utf-8"),
            hashed_password.encode("utf-8")
        )
    except Exception:
        return False
Step 2: JWT Access Token
A JWT (JSON Web Token) is a signed string the server gives to the client after login. The client sends it back in every future request inside the Authorization: Bearer <token> header. The server reads the JWT to know who is making the request — without hitting the database every time.
Setting/Security.py — JWT create & decode
from jose import JWTError, jwt

def create_access_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def decode_access_token(token: str) -> dict | None:
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        return payload   # e.g. {"sub": "5", "exp": 1234567890}
    except JWTError:
        return None     # token expired or tampered with
Step 3: The Full Login Flow
User/Router.py — login endpoint
@UserRouter.post("/login")
async def login(login_data: UserLogin, db: AsyncSession = Depends(get_db)):
    # 1. Find user by username
    user = await user_crud.get_user_by_username(db, login_data.username)
    if not user:
        raise HTTPException(status_code=400, detail="Incorrect username or password")

    # 2. Verify password
    if not verify_password(login_data.password, user.password):
        raise HTTPException(status_code=400, detail="Incorrect username or password")

    # 3. Create JWT access token (short-lived)
    access_token = create_access_token(data={"sub": str(user.user_id)})

    # 4. Create refresh token (long-lived, stored in DB)
    refresh_token_str = generate_refresh_token_string()  # random 32 bytes
    await user_crud.create_refresh_token(db, user.user_id, refresh_token_str)

    return {"access_token": access_token, "refresh_token": refresh_token_str, "token_type": "bearer"}

9. Depends() — Dependency Injection
How FastAPI automatically provides values to endpoints
What is Dependency Injection?
Depends(something) tells FastAPI: "Before calling my function, call something() first, and pass whatever it returns as this parameter." This avoids repeating the same setup code in every endpoint.
three main usages in your project
# 1. Inject a DB session — every endpoint that needs DB
async def some_endpoint(db: AsyncSession = Depends(get_db)):
    ...

# 2. Inject the currently logged-in user
async def some_endpoint(current_user = Depends(get_current_user)):
    ...

# 3. Check permissions (PermissionChecker callable class)
async def some_endpoint(current_user = Depends(PermissionChecker("role", "delete"))):
    ...
get_current_user — Auth on every request
User/Router.py — actual code
async def get_current_user(
    db: AsyncSession = Depends(get_db),
    token: str = Depends(oauth2_scheme)  # extracts Bearer token from header
):
    credentials_exception = HTTPException(status_code=401,
                                            detail="Could not validate credentials")
    if not token: raise credentials_exception

    payload = decode_access_token(token)   # decode JWT
    if payload is None: raise credentials_exception

    user_id = int(payload.get("sub"))      # "sub" = user_id we stored in JWT
    user = await user_crud.get_user_by_id(db, user_id)
    if user is None: raise credentials_exception
    return user                             # passed as current_user to endpoints
Router-level dependency — protects all routes at once

10. PermissionChecker — Role-Based Access
A callable class used as a configurable dependency for admin endpoints
What is a callable class? Class
A callable class is a class with a __call__ method, which makes instances of the class behave like functions. FastAPI calls __call__ for each request, but you can configure behaviour in __init__ once. This is why it's used as a configurable dependency.
User/Router.py — PermissionChecker (actual code)
class PermissionChecker:
    # __init__ runs ONCE when you write: Depends(PermissionChecker("role","update"))
    def __init__(self, feature_name: str, action_name: str):
        self.feature_name = feature_name
        self.action_name  = action_name

    # __call__ runs FOR EVERY REQUEST that uses this dependency
    async def __call__(
        self,
        current_user = Depends(get_current_user),
        db: AsyncSession = Depends(get_db)
    ):
        has_permission = await check_user_permission(
            db, current_user.user_id, self.feature_name, self.action_name
        )
        if not has_permission:
            raise HTTPException(status_code=403, detail="Not enough permissions")
        return current_user
How it is used in Admin/Router.py

12. Alembic — Database Migrations
How the database schema is versioned and updated safely
What is Alembic?
Alembic is a migration tool for SQLAlchemy. When you change your models (add a column, create a table), you create a "migration script". Alembic runs that script to update the actual database without losing existing data. Think of it as Git for your database schema.
commands used in this project
# Auto-generate a migration script from changes in Models.py
alembic revision --autogenerate -m "add refresh_tokens table"

# Apply all pending migrations to the database
alembic upgrade head

# Roll back the last migration
alembic downgrade -1
alembic/env.py — How it reads your models

14. Refresh Token System
Why we have two tokens and how token rotation works
Why Two Tokens? Access vs Refresh Security
Access tokens are short-lived (e.g. 60 min). If stolen they expire fast. Refresh tokens are long-lived and stored in the DB — used only to get a new access token. This way the user stays "logged in" without exposing long-lived credentials on every request.
TokenStored where?ExpiresUsed for
Access TokenFrontend memory/localStorage60 minutesEvery API request (Authorization header)
Refresh TokenDB table refresh_tokensLonger (days)POST /user/refresh — get new access token
Login — returns BOTH tokens
User/Router.py — /login
@UserRouter.post("/login")
async def login(login_data, db = Depends(get_db)):
    user = await user_crud.get_user_by_username(db, login_data.username)
    if not verify_password(login_data.password, user.password):
        raise HTTPException(400, "Incorrect username or password")

    access_token  = create_access_token(data={"sub": str(user.user_id)})
    refresh_token = generate_refresh_token_string()           # random urlsafe 32-byte string
    await user_crud.create_refresh_token(db, user.user_id, refresh_token)
    return {"access_token": access_token, "refresh_token": refresh_token, "token_type": "bearer"}
Token Rotation — POST /user/refresh
Token rotation = when you use a refresh token, the old one is immediately revoked and a brand new one is issued. If someone steals an old refresh token and tries to use it again, ALL sessions for that user get nuked (security!).
User/Router.py — /refresh (token rotation)
@UserRouter.post("/refresh")
async def refresh_token_endpoint(refresh_data, db = Depends(get_db)):
    token_obj = await user_crud.get_refresh_token_by_str(db, refresh_data.refresh_token)
    if token_obj.is_revoked:
        await user_crud.revoke_all_user_refresh_tokens(db, token_obj.user_id)  # nuke all sessions!
        raise HTTPException(401, "Token reuse detected. All sessions invalidated.")

    token_obj.is_revoked = True            # revoke old token
    new_refresh = generate_refresh_token_string()  # create brand new one
    await user_crud.create_refresh_token(db, token_obj.user_id, new_refresh)
    new_access = create_access_token(data={"sub": str(user.user_id)})
    await db.commit()
    return {"access_token": new_access, "refresh_token": new_refresh}
generate_refresh_token_string — Setting/Security.py
actual code
def generate_refresh_token_string() -> str:
    import secrets
    return secrets.token_urlsafe(32)  # cryptographically random, URL-safe string
💡
Why not JWT for refresh? Refresh tokens are stored in DB so we can revoke them at any time. JWTs can't be revoked once issued (stateless). A plain random string stored in DB is perfect for refresh tokens.

15. bcrypt — Password Hashing
How passwords are stored and verified securely — Setting/Security.py
Why hash? Never store plain passwords! Security
If your DB is hacked and passwords are plain text, everyone is exposed. bcrypt is a one-way hashing algorithm. The same password always produces a different hash (because of a random "salt"), so you can't reverse it.
Setting/Security.py — actual code
import bcrypt

def hash_password(password: str) -> str:
    pwd_bytes = password.encode("utf-8")    # str → bytes
    salt = bcrypt.gensalt()              # random salt (makes every hash unique)
    hashed = bcrypt.hashpw(pwd_bytes, salt)  # one-way hash
    return hashed.decode("utf-8")         # bytes → str for DB storage

def verify_password(plain_password: str, hashed_password: str) -> bool:
    try:
        return bcrypt.checkpw(
            plain_password.encode("utf-8"),
            hashed_password.encode("utf-8")
        )  # True if match, False otherwise
    except Exception:
        return False

User/Router.py — actual code
async def check_profile_access(db, current_user, target_user_id) -> bool:
    if current_user.user_id == target_user_id:  # own profile
        return True

    # check if target account is private
    settings = await user_crud.get_active_privacy_settings(db, target_user_id)
    is_private = any(s.privacy_type == "private" and s.is_active for s in settings)
    if not is_private:
        return True  # public account

    # check if current user follows the target
    result = await db.execute(select(UserFollower).where(
        UserFollower.follower_id == current_user.user_id,
        UserFollower.following_id == target_user_id,
        UserFollower.is_active == True
    ))
    if result.scalar_one_or_none() is not None:
        return True  # is a follower

    # check admin bypass
    has_bypass = await check_user_permission(db, current_user.user_id, "user", "view")
    return has_bypass

21. UniqueConstraint & DEFERRED
Advanced DB constraint pattern used in UserBio and PrivacyHistory
The Problem & Solution PostgreSQL
We want only ONE active bio per user at a time. A naive UniqueConstraint(user_id, is_active) would block us from updating: to set a new bio active, we'd first need to deactivate the old one — but deactivating + activating in the same transaction would briefly violate uniqueness mid-way.
⚠️
DEFERRED to the rescue: deferrable=True, initially="DEFERRED" tells PostgreSQL: "Don't check this constraint after every individual statement — check it at the END of the transaction." So you can do both updates in one transaction and it only validates when you COMMIT.
Setting/Models.py — UserBio
class UserBio(Base):
    __tablename__ = "user_bio"
    bio_id   = Column(Integer, primary_key=True)
    user_id  = Column(Integer, ForeignKey("users.user_id"))
    bio_text = Column(Text)
    is_active= Column(Boolean, default=False)
    __table_args__ = (UniqueConstraint(
        "user_id", "is_active",
        name="uq_user_bio_user_active",
        deferrable=True,         # can be deferred
        initially="DEFERRED"      # deferred by default — checked at COMMIT
    ),)
server_default=func.now() — DB-level timestamp
Some models use server_default=func.now() — the database itself sets the timestamp. Others use Python's datetime.utcnow(). Both work, but server_default is more reliable as it uses the DB server's clock.
Setting/Models.py — two approaches
# Approach 1: DB sets the timestamp (PostLike, PostComment, Reel)

# Approach 2: Python sets the timestamp (User, UserRole)
created_at = Column(DateTime)
# ... then in CRUD:

What is a generator? Where is it used in your project?
▼

A generator is a Python function that uses yield instead of return. It pauses at yield, gives a value to the caller, and resumes from that exact point later — unlike a normal function that runs to completion and exits.

In our project: get_db() in Setting/Database.py is the generator:

async def get_db():
    async with SessionLocal() as session:
        yield session    # PAUSE: give session to endpoint
                         # RESUME: session auto-closes

Why generator? It guarantees the DB session always closes after each request — even if an error occurs. This prevents connection leaks.

What is db: AsyncSession? Write the full syntax for creating a session in your project.
▼

AsyncSession is SQLAlchemy's asynchronous database session. It's your open connection to the database — you use it to run queries, add data, commit, and refresh objects — all without blocking the server.

Full creation syntax from our project (Setting/Database.py):

# Step 1: Create async engine (pool of DB connections)
engine = create_async_engine(DATABASE_URL, echo=True)

# Step 2: Create session factory

# Step 3: Generator function yields a session

    async with SessionLocal() as session:
        yield session

# Step 4: Inject via Depends in endpoint
db: AsyncSession = Depends(get_db)

How does JWT authentication work in your project? Explain step by step.
▼
  1. User POSTs credentials to /user/login
  2. Server looks up user in DB, verifies password with bcrypt.checkpw()
  3. Server creates a JWT: jwt.encode({"sub": str(user_id), "exp": ...}, SECRET_KEY, ALGORITHM)
  4. Also creates a random refresh token and saves it in the refresh_tokens DB table
  5. Client stores both tokens in localStorage
  6. Client sends JWT in every request header: Authorization: Bearer <token>
  7. oauth2_scheme extracts the token string; get_current_user decodes it and finds the user
  8. When JWT expires, client sends refresh token to /user/refresh to get a new JWT

Why do we hash passwords? What is bcrypt?
▼

We never store plain text passwords because if the database is hacked, all passwords would be exposed instantly. A hash is a one-way transformation — you cannot reverse it to get the original password.

bcrypt is a password hashing algorithm designed to be computationally slow (to resist brute-force attacks). It includes a random salt so two identical passwords produce different hashes — this prevents rainbow table attacks.

salt = bcrypt.gensalt()           # random salt
hashed = bcrypt.hashpw(pwd, salt)  # one-way hash
ok = bcrypt.checkpw(plain, hashed) # compare at login

What is the difference between db.commit() and db.refresh()?
▼

await db.commit() — saves all staged changes to the database permanently. Before commit, changes exist only in memory (in the session). After commit, they are persisted.

await db.refresh(obj) — reloads the object's attributes from the database into memory. After inserting a new row, the DB auto-generates the primary key (user_id). Without refresh(), the in-memory object still has None for that field.

db.add(new_user)            # stage INSERT
await db.commit()           # save → DB generates user_id
await db.refresh(new_user)  # reload → new_user.user_id is now set!

19
What is expire_on_commit=False in async_sessionmaker?
▼

By default, SQLAlchemy marks all loaded objects as "expired" after a commit(). The next time you access their attributes, it tries to run another SELECT query to refresh them. In async SQLAlchemy, this auto-refresh causes errors because it tries to make implicit DB calls.

expire_on_commit=False disables this: after committing, the in-memory object's attributes remain accessible without triggering another DB query. We manually call await db.refresh(obj) when we need fresh data from the DB.

▼

Motor is the async MongoDB driver for Python. PyMongo is the standard (sync) MongoDB driver.

Since our FastAPI app uses async/await everywhere, we need an async DB driver. Using the sync PyMongo in an async FastAPI app would block the event loop, defeating the purpose of async. Motor wraps PyMongo to provide non-blocking async operations using AsyncIOMotorClient.

mongo_client = AsyncIOMotorClient(MONGO_URL)  # async connection
