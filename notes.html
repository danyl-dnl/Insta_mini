Integration Guide

A beginner-friendly walkthrough of how your frontend talks to your FastAPI backend, with real code examples from your project.

🌐 Chapter 1

What Is Frontend ↔ Backend Integration?

Think of building a web app like running a restaurant. There are two separate worlds:

🖥️
Frontend
The dining room. What the customer sees: menus, tables, décor. Built with HTML, CSS, and JavaScript.
🍳
Backend
The kitchen. Hidden from customers. Stores data, runs logic, returns results. Your FastAPI server.
🛎️
API
The waiter. Carries orders from the dining room to the kitchen and brings back the food (data).
🍽️ The Restaurant Analogy
When you click "Share Post" in your app, JavaScript (the waiter) takes your photo + caption (the order) to FastAPI (the kitchen). FastAPI saves it to the database (cooks it) and sends back { "post_id": 42 } (the meal). JavaScript then updates the screen to show your new post.

In your project, the frontend is a single file — templates/index.html — and the backend is a FastAPI app running at http://127.0.0.1:8001. They communicate using HTTP requests.

What is JSON?

JSON (JavaScript Object Notation) is the language frontend and backend use to talk. It's just structured text that looks like a JavaScript object:

// Backend sends this to the frontend:
{
  "user_id": 3,
  "username": "danyl",
  "email": "danyl@example.com",
  "profile_picture": "https://cdn.example.com/avatar.jpg"
}

Frontend JavaScript reads this and uses each field to update the page — for example, showing the username in the sidebar or the avatar image.

🔧 Chapter 3

HTTP Methods — The 4 Verbs

Every request has a method (verb) that tells the backend what you want to do:

Method What it does Real Example in Your App
GET Fetch/Read data Load all posts: GET /user/posts
POST Create new data Create a post: POST /user/posts
PUT Update existing data Edit your bio: PUT /user/bios/{id}
DELETE Delete data Delete a post: DELETE /user/posts/{id}
💡 Memory Trick Think of CRUD: Create=POST, Read=GET, Update=PUT, Delete=DELETE. Every database operation maps to one of these four verbs.
⚡ Chapter 4

The fetch() Function — How JS Talks to Backend

fetch() is a built-in JavaScript function that sends an HTTP request and gives you back the response. It's the core of all frontend ↔ backend communication.

Simplest Example — GET Request

// Read all posts from the backend
const response = await fetch('http://127.0.0.1:8001/user/posts');
const posts = await response.json();  // convert response to JS array
console.log(posts);  // [{post_id:1, caption:"Hello",...}, ...]

POST Request — Sending Data

// Create a new post
const response = await fetch('http://127.0.0.1:8001/user/posts', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',      // tell server we're sending JSON
    'Authorization': `Bearer ${token}`      // our login token (more on this later)
  },
  body: JSON.stringify({                        // convert JS object → JSON text
    user_id: 3,
    caption: "My first post!",
    image_url: "https://cdn.example.com/img.jpg"
  })
});

const newPost = await response.json();
console.log(newPost);  // {post_id: 42, caption: "My first post!", ...}
⚠️ Important: await fetch() is asynchronous — it doesn't freeze the browser while waiting. The await keyword tells JavaScript to wait for the response before moving to the next line. You must use async function to use await.

Checking If It Worked

const res = await fetch(...);

if (res.ok) {              // res.ok = true if status is 200-299
  const data = await res.json();
  console.log("Success!", data);
} else {
  const error = await res.json();
  console.log("Failed!", error.detail);  // FastAPI puts error in .detail
}
🔁 Chapter 5

Your fetchAPI() Wrapper — The Smart Version

Raw fetch() is repetitive — you'd have to add the base URL and auth token manually every single time. Your project solved this by creating a wrapper function called fetchAPI() that does it all automatically:

index.html — JS
async function fetchAPI(path, options = {}) {
  const token = localStorage.getItem('instagram_access_token');

  const headers = { ...options.headers };

  // Automatically add the JWT token if user is logged in
  if (token) {
    headers['Authorization'] = `Bearer ${token}`;
  }

  // Automatically add Content-Type for JSON bodies
  if (options.body && typeof options.body === 'string') {
    headers['Content-Type'] = 'application/json';
  }

  // Full URL = API_BASE_URL (http://127.0.0.1:8001) + path (/user/posts)
  const res = await fetch(`${API_BASE_URL}${path}`, { ...options, headers });

  // If token expired (401), silently refresh and retry
  if (res.status === 401) {
    await refreshTokens();
    // ... retry the request with the new token
  }
  return res;
}

Why use a wrapper?

❌ Without wrapper (repetitive)
// You write this EVERY SINGLE TIME
const token = localStorage.getItem('instagram_access_token');
await fetch('http://127.0.0.1:8001/user/posts', {
  headers: { 'Authorization': `Bearer ${token}`,
               'Content-Type': 'application/json' }
});
✅ With wrapper (clean)
// Just this. Wrapper handles the rest.
await fetchAPI('/user/posts');
🔐 Chapter 6

JWT Authentication — How Login Works

JWT stands for JSON Web Token. It's like a VIP wristband at a concert — you prove who you are once at the door (login), get a wristband (token), and then show the wristband to enter any area (protected API).

🎟️ Concert Wristband Analogy
Login = Show your ID at the entrance → Get a wristband
Access Token = Your wristband (expires after 30 min — short life)
Refresh Token = A backstage pass to get a new wristband when the old one expires
Bearer Token in Header = Flashing your wristband every time you enter a room

The Two Tokens in Your App

Token Stored In Lifetime Purpose
access_token localStorage['instagram_access_token'] ~30 minutes Sent with every protected API request
refresh_token localStorage['instagram_refresh_token'] 7 days Used to get a new access_token when it expires

How to Use a Token

Every protected route in FastAPI checks for the token in the HTTP Authorization header:

// The header you send with every request:
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
//              ^^^^^^ This keyword is required by the OAuth2 standard
FastAPI — Router.py
# This is how FastAPI verifies the token on the backend:
@UserProtectedRouter.get("/posts")
async def get_all_posts(
    current_user=Depends(get_current_user)  # ← this reads & validates the token
):
    ...
✅ Key Insight If you send a request to UserProtectedRouter without a token (or with an invalid one), FastAPI automatically returns 401 Unauthorized. Your fetchAPI() wrapper catches this and tries to refresh the token silently.
🚪 Chapter 7

The Full Login Flow — Step by Step

  1. User fills the login form

    They type username + password and click "Log In"

  2. JS calls handleLoginUser()

    It sends a POST to /user/login with { username, password } in the body

  3. FastAPI verifies the password

    Checks the hash in the database using verify_password()

  4. Backend returns tokens

    Sends back { access_token: "...", refresh_token: "..." }

  5. JS stores tokens in localStorage

    localStorage.setItem('instagram_access_token', data.access_token)

  6. JS fetches current user profile

    Calls fetchCurrentUser() → GET /user/users/me to load user data + roles

  7. UI updates

    updateCurrentUserUI() shows the user avatar + name in the sidebar. Admin nav appears if the user has admin role.

index.html — handleLoginUser()
async function handleLoginUser(event) {
  event.preventDefault();  // stop the form from reloading the page

  const username = document.getElementById('login-username').value;
  const password = document.getElementById('login-password').value;

  const res = await fetch(`${API_BASE_URL}/user/login`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password })
  });

  const data = await res.json();

  if (!res.ok) throw new Error(data.detail);  // show error if wrong password

  // Store tokens so every future request is authenticated
  localStorage.setItem('instagram_access_token',  data.access_token);
  localStorage.setItem('instagram_refresh_token', data.refresh_token);

  await fetchCurrentUser();  // load user data
  updateCurrentUserUI();      // update sidebar
}
🔄 Chapter 8

Auto Token Refresh — Silent Background Magic

Access tokens expire after ~30 minutes. If the user is still on the page, we don't want to log them out — we silently get a new token in the background.

Request sentfetchAPI()
→
401 receivedtoken expired
→
POST /user/refreshsend refresh_token
→
New tokensstored silently
→
Retry requestwith new token

Your fetchAPI() wrapper handles this automatically. If any request returns 401, it calls refreshTokens(), updates localStorage, and retries the original request. The user never even knows it happened.

Global State — Shared Data Across the App

Since your whole app is in one HTML file, all JavaScript variables are shared globally. Your app uses several global state variables at the top of the script:

index.html — Global Variables
// These variables are accessible from EVERY function in the app
let allUsers = [];          // all users from /user/users
let allPosts = [];          // all posts from /user/posts
let allReels = [];          // all reels from /user/reels
let currentUser = null;     // logged-in user object (or null)
let activeScreen = 'feed'; // which screen is visible
let userStoriesMap = {};    // { user_id: [story, story, ...] }
let profileViewingUser = null; // which profile page is shown
📋 Analogy — A Shared Whiteboard
Imagine the office has one big whiteboard. Anyone can write on it or read from it. allUsers is like the contacts list written there — every function can see it without fetching from the server again. When you call reloadData(), it's like erasing and rewriting the whiteboard from fresh server data.

The reloadData() Function

This is the central function that refreshes all global state by calling multiple backend APIs:

index.html — reloadData()
async function reloadData() {
  // Fetch all users → update global allUsers array
  const userRes = await fetchAPI('/user/users');
  if (userRes.ok) allUsers = await userRes.json();

  // Fetch all posts → update global allPosts array
  const postsRes = await fetchAPI('/user/posts');
  if (postsRes.ok) allPosts = await postsRes.json();

  // Fetch all reels → update global allReels array
  const reelsRes = await fetchAPI('/user/reels');
  if (reelsRes.ok) allReels = await reelsRes.json();
}
🎨 Chapter 10

DOM Manipulation — Updating What You See

The DOM (Document Object Model) is the browser's live representation of your HTML. JavaScript can read and change it to update what the user sees — without reloading the page.

The Key DOM Methods

MethodWhat it doesExample
getElementById(id) Find one element by its id document.getElementById('feed-stream')
element.innerHTML Set or get the HTML inside an element stream.innerHTML = '<p>Hello</p>'
element.textContent Set or get plain text (safer, no HTML) span.textContent = user.username
createElement(tag) Create a new HTML element const card = document.createElement('div')
appendChild(el) Add an element inside another stream.appendChild(card)
classList.add/remove Add or remove a CSS class modal.classList.add('active')

Pattern: Fetch data → Build HTML → Insert into DOM

This is the most common pattern in your app. Here's how the post feed works:

async function renderFeed() {
  const stream = document.getElementById('feed-stream'); // 1. Find container
  stream.innerHTML = '';                                   // 2. Clear old content

  for (const post of allPosts) {  // 3. Loop through the data
    const card = document.createElement('article'); // 4. Create element
    card.className = 'post-card';

    card.innerHTML = `
      <img src="${post.image_url}">
      <p>${post.caption}</p>
    `;  // 5. Fill with data using template literals

    stream.appendChild(card);  // 6. Add to the page
  }
}
💡 Template Literals The backtick (`) syntax lets you embed variables directly in HTML strings using ${variableName}. This is how your app dynamically fills in usernames, images, and captions.

Real Example 1 — Loading the Feed

Here's the full lifecycle of what happens when the app loads and shows you posts:

  1. Page loads → DOMContentLoaded fires

    Browser runs your JS initialization code

  2. fetchCurrentUser() → GET /user/users/me

    If a token exists in localStorage, loads the user's profile + roles

  3. reloadData() → 3 parallel API calls

window.addEventListener('DOMContentLoaded', async () => {

📸 Chapter 13

Real Example 2 — Creating a Post (Form + File Upload)

This is a perfect example of a multi-step frontend→backend flow involving file upload:

User picks imagedrag & drop zone
→
Step 1: Upload filePOST /user/upload
→
Get image URL{ "url": "cdn/img.jpg" }
→
Step 2: Create postPOST /user/posts

Why two steps?

Images can't be sent as JSON (JSON is text-only). So we first upload the file using FormData (multipart), get back a URL, then create the post record with that URL.

index.html — handleCreatePost()
async function handleCreatePost(e) {
  e.preventDefault();

  const file = document.getElementById('post-image-file').files[0];
  const caption = document.getElementById('post-caption').value;

  // ── STEP 1: Upload the image file ──────────────────────
  const formData = new FormData();   // special object for file uploads
  formData.append('file', file);     // attach the file

  const uploadRes = await fetchAPI('/user/upload', {
    method: 'POST',
    body: formData   // NOTE: no Content-Type header — browser sets it automatically
  });
  const uploadData = await uploadRes.json();
  const image_url = uploadData.url;   // e.g. "http://localhost:8001/uploads/abc.jpg"

  // ── STEP 2: Create the post with the image URL ─────────
  const res = await fetchAPI('/user/posts', {
    method: 'POST',
    body: JSON.stringify({
      user_id: currentUser.user_id,
      caption,
      image_url
    })
  });

  if (res.ok) {
    showToast("Shared new post!");  // show success notification
    await reloadData();              // refresh allPosts
    renderFeed();                    // redraw the feed
  }
}
🔍 Chapter 14

Real Example 3 — The Search Feature

Search shows how to use query parameters and debouncing — two very common frontend patterns:

Query Parameters

Query parameters are extra data added to a URL after a ?:

// No query param:  GET /user/posts        → all posts
// With query param: GET /user/search?q=cat → posts matching "cat"

const query = "danyl";
const res = await fetchAPI(`/user/search?q=${encodeURIComponent(query)}`);
//                                       ^ encodeURIComponent escapes special chars

const data = await res.json();
// { users: [{user_id:3, username:"danyl",...}], posts: [...] }

Debouncing — Don't Spam the Server

Without debouncing, every single keypress sends a request. If you type "danyl" (5 chars), that's 5 requests. With debouncing, we wait 350ms after the user stops typing:

let searchTimer = null;

function handleSearchInput(event) {
  const query = event.target.value;

  clearTimeout(searchTimer);    // cancel the previous timer

  searchTimer = setTimeout(() => {
    executeSearch(query);        // only run after 350ms of no typing
  }, 350);
}
⏱️ Debounce Analogy
Imagine an elevator — it doesn't close the doors the instant you step in. It waits a few seconds to see if more people are coming. That wait is "debounce". In search, we wait for the user to stop typing before asking the server.

Real Example 4 — Role-Based UI (Admin Panel)

Some features should only appear for admin users. Your frontend handles this by checking the user's roles after login:

index.html — fetchCurrentUser()
const rolesRes = await fetchAPI(`/user/users/${currentUser.user_id}/roles`);
const roles = await rolesRes.json();

currentUser.isSuperAdmin = roles.includes('SuperAdmin');
currentUser.isAdmin      = roles.includes('Admin') || currentUser.isSuperAdmin;
currentUser.isSupervisor = roles.includes('Supervisor') || currentUser.isAdmin;

Then these flags control what's visible:

// Show Admin nav tab only for admins/supervisors
const adminNav = document.getElementById('nav-admin');
adminNav.style.display = currentUser.isSupervisor ? 'flex' : 'none';

// Show delete button only for admins
const showDelete = currentUser.isAdmin || currentUser.user_id === post.user_id;
⚠️ Frontend vs Backend Security Hiding a button is for UX only — it is NOT security. A clever user can still call your API directly. Real security lives in the backend (PermissionChecker in FastAPI). Frontend checks are just polish.

UI Pattern — Modals (Popup Dialogs)

Modals are popup dialogs that overlay the page. Your app uses the same pattern for every modal: an overlay div with a class toggle.

index.html — HTML structure (same pattern for ALL modals)
<div id="create-post-modal" class="modal-overlay"
     onclick="closeModalOnOverlay(event)">
  <div class="modal-box">
    <header class="modal-header">
      <span>Create New Post</span>
      <button onclick="closeModal('create-post-modal')">&times;</button>
    </header>
    <div class="modal-body">
      <!-- form goes here -->
    </div>
  </div>
</div>
index.html — JS (same functions control ALL modals)
function openModal(modalId) {
  document.getElementById(modalId).classList.add('active');
}
function closeModal(modalId) {
  document.getElementById(modalId).classList.remove('active');
}
// Adding 'active' class makes the overlay visible via CSS:
// .modal-overlay.active { display: flex; }

The same two functions — openModal(id) and closeModal(id) — control every modal in the app just by passing different IDs. This is a great example of reuse.

🍞 Chapter 17

UI Pattern — Toast Notifications

Toasts are small popup messages that appear briefly to inform the user. Your app has one toast element that is reused for every notification:

index.html — showToast()
function showToast(message, isError = false) {
  const toast = document.getElementById('toast-banner');
  document.getElementById('toast-message').textContent = message;

  if (isError) toast.classList.add('error');
  else         toast.classList.remove('error');

  toast.classList.add('show');      // make it visible (CSS transition)

  setTimeout(() => {
    toast.classList.remove('show'); // hide after 3 seconds
  }, 3000);
}

// Usage anywhere in the app:
showToast("Post shared!");           // green success
showToast("Failed to upload", true); // red error

UI Pattern — The Upload Zone

The drag & drop upload zone is a custom replacement for the browser's default file input. It uses a hidden <input type="file"> inside a styled div:

index.html — Upload Zone HTML
<div class="upload-zone" id="uz-post"
     ondragover="uzDragOver(event, 'uz-post')"
     ondrop="uzDrop(event, 'uz-post', 'post-image-file', 'auto')">

  <!-- Hidden native input (invisible, covers the whole zone) -->
  <input type="file" id="post-image-file"
         onchange="uzPreview('uz-post', 'post-image-file', 'auto')">

  <!-- Idle state: icon + instructions -->
  <div class="upload-zone-idle" id="uz-post-idle">
    <i class="fa-solid fa-image"></i>
    <span>Drop your photo here</span>
  </div>

  <!-- Preview state: shown after file is selected -->
  <div class="upload-zone-preview" id="uz-post-preview">
    <img id="uz-post-preview-img">
  </div>
</div>
index.html — uzPreview() — shows image preview after pick
function uzPreview(zoneId, inputId, mediaType) {
  const file = document.getElementById(inputId).files[0];
  const url = URL.createObjectURL(file); // temp local URL for preview

  // Hide the idle placeholder, show the preview
  document.getElementById(`${zoneId}-idle`).style.display = 'none';
  document.getElementById(`${zoneId}-preview`).style.display = 'block';

  // Set the preview image src to the local object URL
  document.getElementById(`${zoneId}-preview-img`).src = url;
}
💡 URL.createObjectURL() This creates a temporary local URL (like blob:http://localhost/abc123) that lets you preview the file immediately without uploading it yet. It only lives as long as the page is open.
